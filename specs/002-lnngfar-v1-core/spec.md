## Clarifications

### Session 2026-03-05

- Q: 当生成目标目录存在冲突文件时，v1 默认策略是什么？ → A: 严格拒绝并退出；检测到任意冲突即失败，提示用户在空目录执行或先清理。
- Q: v1 的 Node.js 最低版本要求是什么？ → A: Node.js >= 18 LTS。
- Q: v1 的 Blueprint 发现范围是什么？ → A: 仅从当前项目本地依赖中发现（含 workspace/link），不扫描全局 npm 包。
- Q: v1 默认将生成产物写入哪里？ → A: 当前工作目录；要求在空目录执行，检测到冲突即按失败处理。
- Q: 执行流程第 5 步“运行 Blueprint 相关测试”在 v1 包含哪些内容？ → A: 仅运行 Blueprint 自身自动化测试。
- Q: `lnngfar-blueprint-cocos` 的项目模板深度应达到什么级别？ → A: 需提供完整小游戏项目骨架（项目配置、资源配置、核心玩法循环、平台适配、UI 与测试），可直接作为小游戏项目起点。

## 用户场景与测试 *(必填)*

### 用户故事 1 - 一键生成 cocos 工程（优先级：P1）

作为项目开发者，我希望执行 `lnngfar cocos` 后可直接得到可打开、可开发、带基础测试的 Cocos Creator 工程脚手架，以便我立刻开始业务开发。

**优先级说明**: 这是 v1 的核心交付能力，若无法稳定生成工程，其他扩展能力均无意义。

**独立测试**: 在全新目录执行 `lnngfar cocos`，验证生成目录结构、入口代码、oops-framework 初始化代码、示例模块与示例测试均存在且可运行。

**验收场景**:

1. **给定** 本机环境满足运行前置条件，**当** 执行 `lnngfar cocos`，**则** 系统生成完整工程并输出成功结果。
2. **给定** 同一输入在同一版本下重复执行，**当** 比较两次生成结果，**则** 生成内容一致且无随机差异。

---

### 用户故事 2 - 第三方 Blueprint 可被规范接入（优先级：P1）

作为 Blueprint 开发者，我希望只要遵守命名与结构规范，就能在不修改 lnngfar 核心代码的前提下被发现、校验与执行。

**优先级说明**: Blueprint 插件化能力是 lnngfar 的架构基石，与生成能力同等关键。

**独立测试**: 准备一个符合规范与一个不符合规范的 Blueprint 包，分别触发加载流程，验证一个成功接入、一个被明确拒绝。

**验收场景**:

1. **给定** Blueprint 包名符合 `lnngfar-blueprint-` 前缀且结构完整，**当** 执行对应命令，**则** Blueprint 被成功发现并执行。
2. **给定** Blueprint 包名不符合前缀规则，**当** 执行对应命令，**则** 系统在校验阶段拒绝执行并返回可理解错误。

---

### 用户故事 3 - 失败可定位且可恢复（优先级：P2）

作为维护者，我希望在任意失败场景下都能立即知道失败阶段与原因，避免静默失败和重复排查。

**优先级说明**: 明确错误反馈直接影响可维护性与团队排障效率。

**独立测试**: 分别制造环境异常、Blueprint 元信息缺失、生成异常、测试失败四类错误，验证输出都包含阶段与原因。

**验收场景**:

1. **给定** 测试阶段出现失败，**当** 执行生成流程，**则** 系统中断流程并明确标注“测试阶段失败”及失败详情。
2. **给定** Blueprint 元信息非法，**当** 执行生成流程，**则** 系统中断并明确标注“Blueprint 校验阶段失败”。

---

### 用户故事 4 - AI 可持续演进（优先级：P3）

作为 AI 辅助开发者，我希望核心与 Blueprint 的职责边界清晰、命名统一、结构稳定，以便在不破坏现有能力的情况下持续扩展功能。

**优先级说明**: 这是 v1 为后续迭代预留的长期价值，优先级低于可用性与稳定性。

**独立测试**: 在不改动核心代码前提下新增最小 Blueprint，并通过规范校验、执行与测试流程，验证扩展路径可行。

**验收场景**:

1. **给定** 新增 Blueprint 遵循约束，**当** 进行接入验证，**则** 无需修改核心代码即可完成扩展。

### 边界情况

- 当运行环境不满足最低 Node.js 版本要求时，系统必须在环境校验阶段失败并给出修复建议。
- 当 CLI 输入未知 Blueprint 名称（例如 `lnngfar unknown`）时，系统必须返回“未发现可用 Blueprint”并列出可用项。
- 当 Blueprint 缺少必需目录（如 `templates/` 或 `tests/`）时，系统必须在校验阶段拒绝执行。
- 当 `blueprint.json` 缺少必填字段时，系统必须指出缺失字段名称并中断流程。
- 当 Blueprint 生成逻辑执行成功但测试失败时，系统必须判定整体失败并退出非零状态码。
- 当生成目标目录存在任意冲突文件时，系统必须严格拒绝并退出，提示用户在空目录执行或先清理后重试。

## 需求 *(必填)*

### 功能性需求

- **FR-001**: 系统必须提供 Node.js CLI 命令 `lnngfar <blueprint>`。
- **FR-002**: v1 系统必须至少支持 `lnngfar cocos` 作为可用命令。
- **FR-003**: 系统必须按固定顺序执行：环境校验 → Blueprint 发现加载 → Blueprint 校验 → 生成执行 → 测试执行 → 结果输出。
- **FR-004**: 系统必须在环境校验阶段验证 Node.js 版本满足 `>= 18 LTS`，不满足时中断并返回失败状态。
- **FR-005**: 系统必须仅接受包名以 `lnngfar-blueprint-` 为前缀的 Blueprint。
- **FR-006**: 系统必须将 Blueprint 逻辑名称解析为包名前缀后的后缀部分，并用于 CLI 映射。
- **FR-007**: 系统必须校验每个 Blueprint 至少包含 `blueprint.json`、`templates/`、`generators/`、`tests/`、`README.md`。
- **FR-008**: 系统必须校验 `blueprint.json` 包含 `name`、`packageName`、`version`、`description`、`target`、`language`、`engine`、`testFramework`。
- **FR-009**: 系统必须拒绝不符合命名规则或元信息规则的 Blueprint，并禁止进入生成阶段。
- **FR-010**: 系统必须内置并可执行 `lnngfar-blueprint-cocos`，用于生成基于 oops-framework 的 Cocos Creator 工程脚手架。
- **FR-011**: `lnngfar-blueprint-cocos` 生成结果必须包含可被 Cocos Creator Dashboard 直接导入的工程骨架：Creator 版本声明、项目设置、启动场景与入口脚本。
- **FR-025**: `lnngfar-blueprint-cocos` 生成结果必须至少包含以下关键路径：`README.md`、`package.json`、`tsconfig.json`、`settings/v2/packages/project.json`、`settings/v2/packages/program.json`、`settings/v2/packages/builder.json`、`settings/v2/packages/engine.json`、`settings/v2/packages/device.json`、`assets/main.scene`、`assets/main.scene.meta`、`assets/script/Main.ts`、`assets/script/Main.ts.meta`、`assets/script.meta`。
- **FR-012**: 系统必须保证生成过程确定性：相同输入在相同版本下输出一致。
- **FR-013**: 系统必须在生成完成后执行 Blueprint 自身自动化测试（v1 默认不在该步骤执行生成工程内的示例测试）。
- **FR-014**: 当任一测试失败时，系统必须立即中断并返回失败状态。
- **FR-015**: 在任意失败场景下，系统必须输出失败阶段（环境/Blueprint/生成/测试）与可理解错误信息。
- **FR-016**: 系统必须禁止静默失败；所有失败都必须写入 `stderr`，并包含可追踪错误码。
- **FR-017**: 系统架构必须支持第三方在不修改核心代码前提下新增 Blueprint。
- **FR-018**: Blueprint 升级不得要求同步修改核心流程即可维持兼容执行。
- **FR-019**: lnngfar 核心模块必须具备自动化测试，且核心测试可独立运行。
- **FR-020**: 每个 Blueprint 必须自带自动化测试，且生成工程必须包含可运行的示例测试。
- **FR-021**: 当生成目标目录存在任意冲突文件或目录时，系统必须终止生成并返回失败状态，不得执行覆盖写入。
- **FR-022**: FR-021 触发时，系统必须输出明确修复建议（在空目录执行或先清理冲突文件后重试）。
- **FR-023**: Blueprint 发现阶段必须仅扫描当前项目本地依赖（含 workspace/link），不得将全局 npm 包纳入发现结果。
- **FR-024**: 在未提供额外路径参数时，生成产物默认写入当前工作目录。

### 关键实体 *(功能涉及数据时填写)*

- **Blueprint 包**: 可被 lnngfar 发现和执行的插件单元，关键属性包括 `packageName`、逻辑名称、版本、目录结构与元信息完整性。
- **Blueprint 元信息（blueprint.json）**: 描述 Blueprint 身份与能力的规范对象，关键属性包括名称、目标技术栈、语言、引擎、测试框架。
- **生成会话**: 一次 `lnngfar <blueprint>` 执行过程，关键属性包括输入参数、执行阶段、阶段结果、错误信息与最终状态。
- **生成产物工程**: Blueprint 输出的项目脚手架，关键属性包括目录结构、入口代码、初始化代码、示例模块与示例测试。
