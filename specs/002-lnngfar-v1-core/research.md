# Phase 0 调研记录：lnngfar v1

## 决策 1：核心实现语言与运行时

**决策**  
核心使用 TypeScript 5.x，运行时基线 Node.js `>=18 LTS`。

**理由**  
TypeScript 提供更清晰的类型边界，适合 Blueprint 契约与阶段状态建模；Node.js 18 LTS 满足稳定性与兼容性要求。

**备选方案**
- 纯 JavaScript：实现更快，但契约边界更易漂移。
- Node.js 20+ 强制：更现代，但与规格中已确认的 `>=18 LTS` 不一致。

## 决策 2：Blueprint 发现策略

**决策**  
仅扫描当前项目本地依赖（含 workspace/link），不扫描全局 npm 包。

**理由**  
保证可复现性，避免“同命令在不同机器发现不同 Blueprint”的隐式行为。

**备选方案**
- 本地 + 全局混合扫描：灵活但不可控。
- 仅全局扫描：对仓库内开发体验差。

## 决策 3：Blueprint 元信息校验方式

**决策**  
使用 JSON Schema（`contracts/blueprint.schema.json`）+ `ajv` 执行运行时校验。

**理由**  
可将规范外显为机器可校验契约，便于第三方 Blueprint 接入与 CI 自动验证。

**备选方案**
- 手写 if/else 校验：分散且难维护。
- 仅 TypeScript 类型：运行时无法直接保护外部 JSON 输入。

## 决策 4：冲突文件策略

**决策**  
目标目录存在任意冲突文件/目录时立即失败退出，不执行覆盖写入。

**理由**  
与规格澄清一致，最小化破坏性行为并提升可预测性。

**备选方案**
- 默认覆盖：风险高，易误伤用户文件。
- 白名单覆盖：实现复杂且规则解释成本高。

## 决策 5：第 5 阶段测试范围

**决策**  
第 5 阶段仅执行 Blueprint 自身自动化测试；生成工程示例测试作为补充验证路径保留在 quickstart。

**理由**  
符合规格澄清，可在 v1 保持执行流程简洁并控制命令耗时。

**备选方案**
- 同时执行生成工程测试：质量更强，但环境要求更高、耗时更长。
- 仅检查测试文件存在：无法保证可运行性。

## 决策 6：确定性生成实现约束

**决策**  
模板渲染与文件写入采用稳定顺序（排序后写入），禁用随机名与隐式时间戳注入。

**理由**  
直接支持 SC-002（20 次一致性 100%）的可验证目标。

**备选方案**
- 允许时间戳：会破坏输出一致性。
- 仅做最终 diff 过滤：实现复杂且不可解释。

## 澄清项收敛结论

本阶段无未解决澄清项，已可进入任务拆解阶段。
